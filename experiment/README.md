## 二、项目背景与研究意义

### 2.1 研究背景

大语言模型（Large Language Model, LLM）的部署正在从云端向端侧迁移。端侧部署具有数据隐私性好、响应延迟低、无需网络连接等优势，已成为学术界和工业界的重要研究方向。Apple Silicon 设备（如 M4 芯片）凭借其统一的内存架构和高效的 Neural Engine / GPU 计算单元，成为端侧推理的重要平台之一。

llama.cpp 是目前最为成熟和广泛使用的端侧 LLM 推理框架之一，支持多种模型架构、多种量化格式、多种硬件后端以及高并发服务。然而，llama.cpp 作为一个通用框架，其默认配置并未针对特定硬件平台（如 M4）进行深度优化。在实际使用中，用户常常面临以下问题：

- 长上下文场景下 KV cache 占用大量内存，导致模型无法加载或运行缓慢；
- 首 token 生成延迟（Time To First Token, TTFT）过高，影响交互体验；
- 单 token 生成延迟（Time Per Output Token, TPOT）受限于后端算子效率；
- 多请求并发时吞吐量下降明显，server 调度策略有待优化。

### 2.2 研究意义

本课题以 MacBook Air M4 为研究平台，以 llama.cpp 为研究对象，通过源码级分析深入理解端侧推理系统的性能瓶颈，并通过针对性优化提升其端到端推理性能。研究成果不仅可为 Apple Silicon 设备上的 llama.cpp 部署提供优化参考，也能加深研究者对以下核心问题的理解：

- 大模型推理中的计算图构建、算子调度与内存管理；
- 模型量化与 KV cache 量化的原理、实现及其对速度、内存与质量的影响；
- 端侧推理框架在不同负载下的性能行为与优化空间。

## 三、研究目标

本课题的总体目标是：

> **在 MacBook Air M4 的资源约束下，系统分析 llama.cpp 的端到端推理性能瓶颈，并通过源码级优化使 Gemma-3 系列模型在该平台上获得接近最优的推理性能。**

具体目标包括：

1. 建立一套针对 M4 平台的 llama.cpp 性能测量与分析方法；
2. 识别 `llama-cli` 同步推理和 `llama-server` 并发服务中的关键性能瓶颈；
3. 研究不同量化策略（权重量化、KV cache 量化）对速度、内存和输出质量的影响；
4. 针对发现的瓶颈，在源码层面实现至少一项针对性优化，并验证其效果；
5. 形成面向 M4 平台的 llama.cpp 推理优化结论与建议。

## 四、研究内容

### 4.1 性能测量与基线建立

- 选取合适的测试模型（Gemma-3 系列）和测试输入（短 prompt、长 prompt、多轮对话等）；
- 建立可复现的性能测试流程，统一测量指标：
  - 模型加载时间；
  - 首 token 延迟（TTFT）；
  - 单 token 延迟（TPOT）；
  - 吞吐量（tokens/s）；
  - 峰值内存占用（RSS）；
  - 输出质量指标（perplexity、任务准确率等）。
- 在默认配置下建立性能基线。

### 4.2 推理框架层分析

- 分析 `llama-cli` 的同步推理流程：模型加载、tokenize、batch 构建、`llama_decode` 调用、sampling、输出生成；
- 分析 `llama-server` 的并发处理机制：slot 管理、continuous batching、KV cache 分配与复用、请求调度；
- 研究 batch size、ubatch size、context size、线程数等参数对性能的影响。

### 4.3 量化策略研究

- 研究不同权重量化格式（如 F16、Q8_0、Q4_K_M、Q4_0 等）对模型大小、加载内存、推理速度和输出质量的影响；
- 研究不同 KV cache 量化格式（如 f16、q8_0、q4_0 等）对长上下文推理的内存占用和速度的影响；
- 分析权重量化与 KV cache 量化的组合效果，寻找最佳搭配。

### 4.4 算子与后端分析

- 分析 llama.cpp 在 Apple Silicon 上的后端选择机制（CPU / Metal GPU）；
- 研究 Flash Attention、线程调度、内存分配策略等算子/后端相关参数的影响；
- 必要时在关键路径添加性能插桩，精确测量各环节耗时。

### 4.5 源码级优化

基于前述分析，在源码层面实现至少一项针对性优化。可选方向包括但不限于：

- 在关键路径添加更细粒度的性能观测与统计；
- 优化 `llama-server` 的请求调度或 KV cache 管理策略；
- 改进 batch 构建或 sampling 流程以降低延迟；
- 针对 M4 平台调整后端算子调度或内存分配策略。

### 4.6 验证与总结

- 通过对比实验验证优化前后的性能差异；
- 分析优化有效或无效的原因；
- 总结面向 M4 平台的 llama.cpp 推理优化经验。

## 五、技术路线

本课题采用“测量—分析—假设—干预—验证”的闭环研究方法：

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│  建立基线   │ → │  性能测量   │ → │  瓶颈定位   │
└─────────────┘    └─────────────┘    └──────┬──────┘
                                             ↓
┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│  总结结论   │ ← │  效果验证   │ ← │  源码级优化 │
└─────────────┘    └─────────────┘    └─────────────┘
```

### 5.1 关键步骤

1. **环境搭建与工具熟悉**：编译 llama.cpp，熟悉 `llama-cli`、`llama-server` 的基本用法和参数含义；
2. **测量体系建设**：设计测试用例，编写日志解析与指标统计工具；
3. **基线测试**：在默认配置下完成全面性能测试；
4. **单变量分析**：逐一改变关键参数，观察性能变化；
5. **瓶颈定位**：结合源码与测量数据，确定主要瓶颈所在；
6. **优化实现**：针对瓶颈提出假设并修改源码；
7. **对比验证**：在相同测试条件下对比优化前后性能；
8. **报告撰写**：整理数据、分析原因、形成结论。

## 九、注意事项

1. **源码修改应以学习为目的**：所有修改需基于对源码的深入理解，避免盲目试错。
2. **每次修改需有明确假设**：优化前应提出清晰的性能假设，优化后必须通过测量验证或证伪。
3. **保持可复现性**：所有实验应记录完整的命令、参数、环境版本和原始日志。
4. **遵守开源协议**：llama.cpp 采用 MIT 协议，源码修改与学习应遵守相关许可。
5. **不提交 AI 生成的代码**：本课题是个人学习研究，所有源码修改应由研究者本人理解、实现并负责。
