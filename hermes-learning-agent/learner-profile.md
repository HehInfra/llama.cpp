# Learner Profile

本文件记录学习起点。后续学习代理应根据这里的画像决定探索深度、解释粒度和补课顺序。

## 当前状态

以下是初始画像，等待用户确认或修改：

- 用户希望在 llama.cpp 仓库中启动一个透明、可复盘的源码学习代理工作流。
- 不能假设用户已经熟悉 llama.cpp 的整体架构、GGML/GGUF、推理流程、量化、后端抽象或性能优化细节。
- 可以假设用户愿意借助 AI 阅读代码，但最终希望自己能讲清路径、证据和不确定点。
- C++ 语言学习是明确目标之一：阅读源码时要同步解释 C++ 语法、类型、对象生命周期、模板、宏、指针/引用、RAII、内存管理、并发相关写法和工程惯用法。
- CMake、Shell、Python 工具脚本、平台后端等也可能需要随源码做最小补课，但优先级低于 C++ 主线。
- 对大型开源仓库的阅读方法仍需要被显式示范：如何从 README、目录、示例、测试、构建系统和调用链中建立方位感。

## 学习偏好

- 希望看到一个学习者如何逐步探索，而不是直接得到专家式总结。
- 可以接受 agent 自主选择路径，但要求每次说明为什么选择当前问题。
- 更适合小步推进：一次回答一个具体问题，逐渐串成整体图景。
- 希望保留猜测、误解、修正和仍未理解的地方。
- 希望通过真实源码同时学习架构拆解方法、C++ 阅读能力、工程组织、调试方法和 AI 推理系统基础。
- 更偏向架构理解：优先建立模块边界、数据流、调用链、抽象层次和设计取舍，再进入局部实现细节。

## 解释粒度

学习代理默认应使用以下粒度：

- 首次遇到关键术语时，给出能继续读代码的最小解释。
- 先用当前代码上下文解释术语，再补充背景知识。
- 对 C++ 写法不要默认用户已经懂；如果影响理解，应解释“语法长什么样、在这里是什么意思、为什么这样写、读代码时该抓住什么”。
- 每轮探索至少记录一个与当前源码相关的 C++ 语言学习点；如果本轮确实没有新语言点，要说明原因。
- 对暂时不阻塞的问题，记录到概念债务或语言债务，不要无限展开。
- 所有解释都要服务于当前源码问题，不要脱离仓库写成通用教程。
